The Ink Path
Field Notes Edition
Field Engineering · reMarkable 2

Ink at eight milliseconds.

pi-collab was built on a reasonable assumption: a third-party app can never match the notebook's pen latency, because xochitl keeps its optimized stack to itself and everyone else goes through sockets. riddle now matches it — over a socket. The assumption named the wrong suspect.

By Claude (Fable 5)
Subject pi-collab vs. riddle
Corpus two codebases, one tablet
Filed July 7, 2026

I.Two apps, one assumption

Both apps live on the same reMarkable 2, both were written in Rust by the same author's sessions, and both read the pen the same way: straight off the Wacom digitizer as raw evdev, full 4096-level pressure, axes swapped and inverted with the exact transform the community reverse-engineered years ago. pi-collab is a handwriting chat — you write in a strip at the bottom, tap SEND, and your ink rides to a headless pi agent as a PNG. riddle is an enchanted diary — you write, rest the pen, and the page drinks your ink before answering in a flowing hand. Up to the moment a pixel needs to reach glass, they are siblings.

After that moment they part company completely. When pi-collab was built, the ceiling seemed obvious: reMarkable's own notebook draws through xochitl's in-process engine, while an AppLoad app draws through a shared framebuffer and a socket into someone else's render loop. Sockets versus the vendor's private fast path — surely no contest. That framing survived until riddle's takeover mode landed and matched the notebook stroke for stroke, without abandoning the socket.

II.The machine underneath

To see why the socket was framed, you have to see what this tablet actually runs. The reMarkable 2 is an NXP i.MX7Dual — two Cortex-A7 cores, about 1 GB of RAM — and, crucially, a SoC with no e-ink controller block. Its predecessor's i.MX6 SoloLite had a hardware EPDC that turned pixel buffers into panel waveforms on its own; the rM2's panel hangs off a plain parallel display interface instead. Every waveform this device has ever drawn was computed by ordinary CPU threads. The "display controller" is a program — which is the single fact this whole story turns on, because programs can be relocated, reimplemented, or borrowed.

Above the silicon sits a Yocto-built embedded Linux — kernel 5.4.70 on this unit, systemd as init (with logind handling power), a BusyBox userland thin enough that timeout doesn't exist and head rejects its own common flags, and glibc underneath it all. There is no display server. No X11, no Wayland, no compositor arbitrating windows — nothing between an application and /dev/fb0 but intent. Input is raw kernel evdev: event0 is the power key, event1 the Wacom I2C pen digitizer (hover, tilt, 4096 pressure levels at hardware rate), event2 the capacitive touch panel. Any root process may open them; the "input stack" is a read() loop.

Into that vacuum reMarkable ships exactly one program: xochitl, a monolithic Qt 6/QML application that is simultaneously the document library, the notebook editor, the cloud sync client, the Wi-Fi settings panel, the input reader — and, on this device, the display driver itself, hosting the SWTCON threads that drive the panel. It is not an app on a desktop; it is the desktop, the window system, and the graphics card's firmware, fused. Since OS 3.20 one load-bearing organ of it lives in a separate scene-graph plugin, libqsgepaper.so — a detail that becomes the entire plot of section V.

The disk layout explains how a hacked device survives: two small A/B root partitions (~256 MB, swapped wholesale on OTA updates — ours sits 91% full) and a 6.4 GB /home that persists. Everything the community adds lives in /home/root; everything reMarkable owns gets replaced under it. The modern mod stack rides that line: xovi injects extension .sos into xochitl at launch, qt-resource-rebuilder patches its QML resources in memory, and AppLoad adds a launcher whose qtfb backend rents guest apps a shared-memory framebuffer inside xochitl's own scene. That rented window is where pi-collab lives — and the rent, it turns out, is paid in milliseconds.

III.Anatomy of the middleman

The windowed path both apps started on is AppLoad's qtfb: the app draws RGB565 into a shared-memory buffer, then sends a small update message over a Unix socket. Inside xochitl, a plugin marks a Qt scene item dirty; Qt repaints the buffer into its scene on its own schedule; xochitl's e-ink backend then selects a waveform and queues the region for the panel. Every hop is cheap. The scheduling between hops is not.

What makes the assumption interesting is that pi-collab did everything right on its side of the socket. It coalesces ink flushes to 12 ms — more aggressive than riddle's original 35. It requests the UltraFast waveform; its own source calls the refresh mode "THE latency lever." It rejects palms, reads the digitizer directly, throttles the big conversation region separately (110 ms) so it never starves the ink. None of it was enough, because the lever that matters sits on the far side of the socket. When riddle's windowed build dropped its flush interval from 35 ms to 15, the author's on-device verdict came back unmoved: hmmm quite slow still…

The socket crossing itself is microseconds. The milliseconds live on the other side, in a render loop nobody outside xochitl controls.

IV.What the notebook knows

xochitl's own canvas never negotiates with a scene graph for its ink. Because the rM2 has no hardware e-ink controller, a software TCON drives the panel from CPU threads — a generator computing waveform phases against a per-temperature table, a vsync thread feeding the panel. Pen strokes get a dedicated DU (direct-update, near-binary) waveform in a priority mode that jumps the queue. That is the entire moat: not a faster transport, but a private engine with a private scheduler.

For years that engine was compiled into the xochitl binary itself. The original rm2fb project chased its function addresses release by release and stalled at OS 3.3 in early 2024 — which is why takeover looked dead for anyone on modern firmware. Then reMarkable did the community an accidental favor: from OS 3.20 the engine moved into libqsgepaper.so — a plugin, with exported symbols: EPFramebufferSwtcon::initialize, ::update, ::shutdown. The same library riddle's quill backend interposes on the Paper Pro. The moat became a loadable library.

V.The heist

The instrument is timower/rM2-stuff's rewritten rm2fb server. It dlopens the vendor plugin and calls its exported initializer — hooking malloc/calloc for the duration so the engine's framebuffer allocation lands in shared memory (/dev/shm/swtfb.01) where clients can draw. Three unexported internals (lock, update, unlock) are resolved as offsets from the exported ::update symbol, keyed by the library's GNU build ID. On this tablet — OS 3.24.0.149, past anything in the project's README — the build ID extracted over SSH matched the 3.23.0.54 entry byte for byte. The prebuilt release binary worked untouched, first try.

riddle's takeover script stops xochitl (with a trap that always restarts it), hosts that server from its own app folder, and then speaks the plainest protocol of this whole story: one raw 32-byte UpdateParams struct per dirty region, over a Unix stream socket, answered by a single bool byte. The ack makes every update synchronous — backpressure for free, no client-side coalescing needed beyond an 8 ms cadence. Ink is sent as DU with the priority flag, which the server maps onto xochitl's own pen mode. It is, pointedly, still sockets.

Measured on-device with a bar-sweep diagnostic: 24 fast updates in 921 ms, of which 720 ms were deliberate sleeps — roughly eight milliseconds per synchronous update round-trip, engine kick included. The author's verdict this time: AMAZING this works.

VI.The bill

Nothing here is free. Takeover is exclusive: xochitl is stopped for exactly as long as the diary is open, so the app must own everything it touched — a five-finger tap to exit, the power button remapped to a sleep page (with the kernel's suspend_stats counter as the authoritative "we actually slept" signal, and a Wi-Fi re-association nudge on wake), a systemd transient unit so the launcher's death can't orphan the session, and a restore trap for every exit path. It is also coupled: those three function offsets are keyed to a build ID, so a future OS update can orphan takeover mode until rM2-stuff maps the new library — while the windowed path, for all its lag, survives updates untouched. Both modes ship in the same binary; the launch environment picks.

And the assumption? It was half right. xochitl's stack is optimized, and no amount of client-side tuning reaches it from a window. The half that was wrong is that sockets were ever the reason. pi-collab's pen plumbing is already identical to riddle's; the same server sits one dlopen away. Whether a chat app wants to own the whole panel is a design question now — not a physics one.
